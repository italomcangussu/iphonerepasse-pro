# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Project overview

iPhoneRepasse Pro is a React 19 + Vite SPA for managing a used-iPhone resale business (inventory, point-of-sale, finance, warranties) plus an embedded WhatsApp/Instagram **CRM Plus** with an AI agent. The backend is Supabase (Postgres + Auth + Edge Functions written in Deno). The app ships as an installable PWA. UI text and domain language are Brazilian Portuguese.

## Commands

```bash
npm run dev          # Vite dev server on :3000
npm run build        # Production build to dist/
npm run preview      # Preview the built bundle
npm start            # Serve dist/ statically on :3000 (production-like, via `serve`)
npm run lint         # ESLint (flat config)
npm run typecheck    # tsc --noEmit

npm test             # Vitest (watch) — frontend jsdom tests
npm run test:run     # Vitest single run (CI)
npm run test:deno    # Deno tests for supabase/functions/** (.test.ts / .deno.ts)
```

Run a single frontend test file or filter:
```bash
npx vitest run pages/Inventory.test.tsx
npx vitest run -t "reserves stock"      # by test name
```
Edge-function (Deno) tests are excluded from Vitest. There is also a Node-environment Vitest config for a subset: `npx vitest --config vitest.supabase.config.ts`.

Smoke tests (Playwright + migration/severity checks): `npm run smoke:run` (or `smoke:test`, `smoke:migrations`, `smoke:severity` individually).

## Environment

`.env.local` must define `VITE_SUPABASE_URL` and `VITE_SUPABASE_ANON_KEY` (see [services/supabase.ts](services/supabase.ts)). `GEMINI_API_KEY` is wired into `process.env.API_KEY` by Vite. Push/PWA and CRM origin vars are in [.env.example](.env.example). Edge functions read their own secrets from `supabase/functions/.env`.

## Architecture

### Two apps, one bundle
The same SPA serves two products, selected at runtime in [App.tsx](App.tsx) by hostname/hash:
- **Main ERP app** — full router under `<DataProvider>` with all the business pages.
- **CRM Plus standalone** — when the host is the CRM hostname (`crm.iphonerepasse.com.br`, overridable via `VITE_CRM_HOSTNAME`) or the hash starts with `#/crmplus`, it renders [CRMStandaloneApp](components/crm/CRMStandaloneApp.tsx) instead. The standalone app **does** mount its own `<DataProvider>` (the CRM is unified with the ERP data layer now — see [useCRMStore.ts](components/crm/useCRMStore.ts)), wraps a `BrowserRouter` (clean paths, not hashes), and shares the same `Login` + `ProtectedRoute`. Host/path/branding logic lives in [lib/crmRouting.ts](lib/crmRouting.ts) and [lib/runtimeBranding.ts](lib/runtimeBranding.ts) (swaps favicons, manifest, theme color, title per product).

The main ERP app uses `HashRouter`; the CRM standalone app uses `BrowserRouter`. CRM pages are also mounted inside the main app under `/crm/*`.

### State & data layer
- [services/dataContext.tsx](services/dataContext.tsx) is the heart of the ERP: a single large `DataProvider` that loads stock, customers, sellers, sales, debts, finance, parts, reservations, etc. from Supabase and exposes every mutation as an async action. Most ERP pages consume this one context. When adding a business entity, you extend this context, its `DataContextType` interface, and [types.ts](types.ts).
- [contexts/AuthContext.tsx](contexts/AuthContext.tsx) — Supabase Auth session + `user_profiles` row. Distinguishes the DB base role (`admin`/`seller`) from an operational `AppRole` (`admin`/`manager`/`seller`) read from user metadata.
- CRM Plus reads from the same `DataProvider`. [components/crm/useCRMStore.ts](components/crm/useCRMStore.ts) (`CRMStoreProvider`) is now a thin wrapper over `useData()` that just resolves the active `StoreLocation` (kept for backward compatibility — store selection is unified). CRM pages otherwise talk to Supabase directly: reads/writes against `crm_*` tables (`crm_conversations`, `crm_messages`, `crm_leads`, `crm_channels`, `crm_filter_views`, …) and invoke the `crm-*` edge functions for sends/actions.

### Permissions
Two layers gate the UI:
- **Route roles** — `<ProtectedRoute allowedRoles={['admin']}>` (most `/crm/*` admin pages).
- **Permission matrix** — `<ProtectedRoute requiredPermission="inventory">`. Keys, labels, route prefixes and per-role defaults (`visible`/`editable`/`deletable`) are defined in [lib/permissions.ts](lib/permissions.ts) and surfaced through [contexts/PermissionsContext.tsx](contexts/PermissionsContext.tsx). Adding a gated page means adding a `PermissionKey` here.

### Supabase Edge Functions (`supabase/functions/`)
Deno functions, one folder per function with `index.ts`. Shared code lives in [supabase/functions/_shared/](supabase/functions/_shared/) (`crm.ts` helpers, `uazapi.ts` WhatsApp adapter, the AI routing/entry/payload engines). These are **excluded from tsconfig, ESLint, and Vitest** — they are typechecked/tested via Deno.

The CRM AI pipeline is the most intricate part:
- Inbound WhatsApp/Instagram webhooks → `crm-uaz-webhook-receiver` / `crm-instagram-webhook-receiver` → `crm-ai-inbound`.
- Routing decision (AI vs human) is computed in [_shared/crm_ai_routing.ts](supabase/functions/_shared/crm_ai_routing.ts) from channel `ai_entry_mode` + store `fallback_mode` + webhook config.
- **AI→human handoff has two distinct states** (see the header comment in [crm-ai-inbound/index.ts](supabase/functions/crm-ai-inbound/index.ts)): `transferencia_pendente` (AI stopped, lead blinks/locks at top of list, no human yet) vs `em_atendimento_humano` (a human clicked "Assumir"). Keep these in sync with `ConversationsPage`. Never route a handoff back through `applyAiRoutingDecision`.

### n8n integration
The AI agent's conversational logic runs in an external n8n workflow. [scripts/n8n/](scripts/n8n/) holds the deterministic core, memory guardrails, reply-context builders, and a scenario harness/quality-gate test suite (run the `test-repasse-*.mjs` scripts with `node`). The deployed workflow is fragile: prefer surgical patches over the build script (which clobbers and leaves the workflow OFF), and always reactivate after deploy. The test harness debounces by WhatsApp JID, so sandbox scenarios must use unique JIDs or they collide.

**Connection (two HTTP directions, no shared DB):**

- **App → n8n (inbound trigger):** when AI should answer, [_shared/crm_ai_inbound_dispatch.ts](supabase/functions/_shared/crm_ai_inbound_dispatch.ts) builds a compact payload via `buildCompactAiInboundPayload` ([_shared/crm_ai_payload.ts](supabase/functions/_shared/crm_ai_payload.ts)) and `POST`s it to the **per-channel** `ai_resume_webhook_url` (the n8n Webhook trigger node URL, stored on the channel row — there is no global webhook env var). The call has a 15s timeout (`DISPATCH_FETCH_TIMEOUT_MS`); raw inbound is truncated to 64 KB; every attempt is logged to `crm_event_log` as `crm_ai_inbound_dispatched` (with `webhook_url_host`, status, error).
- **n8n → app (callbacks):** n8n calls the [crm-n8n-api](supabase/functions/crm-n8n-api/index.ts) edge function, authenticated by the `x-api-key` header matching the `CRM_N8N_API_KEY` secret (falls back to a Supabase Bearer token). It dispatches on a `{ action, storeId, payload }` body; supported actions: `publish_event`, `upsert_lead`, `schedule_message`, `get_statistics`, `prepare_broadcast`, `broadcast_stats`, `cancel_broadcast`, `sync_campaign_tags`, `test_webhook`. Outbound replies themselves go through the WhatsApp/Instagram send path (`crm-send-message` / `uazapi.ts`), not this API.
- **Workflows:** the live workflow is `Cr4fPWe0prwS6XjI` ("ia repasse-pro v2 avancada"). The legacy `ia repasse-pro` workflow (`oWNdWPUq6kEFitsnl8OpH`) is no longer the operational reference. [scripts/n8n/export-repasse-workflow.mjs](scripts/n8n/export-repasse-workflow.mjs) exports the live v2 workflow to `output/n8n/ia-repasse-pro-v2-current.json`; see [docs/superpowers/specs/2026-06-12-n8n-repasse-v2-live-context.md](docs/superpowers/specs/2026-06-12-n8n-repasse-v2-live-context.md) for the current n8n context. n8n API base/key come from env read by the scripts — the real keys are `N8N_API_KEY` + `N8N_BASE_URL` (its origin is the n8n host; some older scripts wrongly expect `N8N_PUBLIC_API`/`N8N_MCP_URL`). Never edit the deployed workflow blindly; export, patch, validate ([validate-repasse-next-workflow.mjs](scripts/n8n/validate-repasse-next-workflow.mjs)), then reactivate. **Anti-regression guard (run FIRST, every time you touch the live workflow):** [guard-live-workflow-sync.mjs](scripts/n8n/guard-live-workflow-sync.mjs) GETs the live workflow, recognizes whether it was edited *manually* since the project's last sync, and — on drift — re-exports the snapshot and re-syncs the byte-exact `.js` mirrors (`repasse-code-*.js`, auto-discovered) so you never patch/analyze against a stale base. For Claude Code this fires **automatically** via the `PreToolUse` Bash hook in [.claude/settings.json](.claude/settings.json) → [scripts/n8n/hooks/n8n-live-guard-hook.mjs](scripts/n8n/hooks/n8n-live-guard-hook.mjs). For Codex, the same non-blocking hook is registered in [.codex/hooks.json](.codex/hooks.json); review/trust it with `/hooks` on first use. Other agents without hook support must run `node scripts/n8n/guard-live-workflow-sync.mjs` manually first — see [AGENTS.md](AGENTS.md). The hook syncs and injects a "re-read the files" warning when the live version moved. Guard state/reports live under `output/n8n/.live-guard/`; `--check` exits 3 on drift without writing, `--json` for machine output. Surgical patch scripts (e.g. [patch-parse-memory-tradein-state.mjs](scripts/n8n/patch-parse-memory-tradein-state.mjs)) follow the same shape: GET → backup under `output/n8n/backups/` → exact-string `.replace()` with guards → `new Function()` syntax-assert → PUT → `/activate` → re-export. `DRY=1` previews against live without writing. **Decomposition tool (node-by-node maintainability):** [scripts/n8n/repasse-maint.mjs](scripts/n8n/repasse-maint.mjs) (`pull`/`status`/`build`/`deploy [--confirm]`) mirrors the live workflow into the versioned [n8n/ia-repasse-pro-v2/](n8n/ia-repasse-pro-v2/) (canonical `workflow.json`, decomposed `nodes/code/*.js` + `nodes/prompts/*.md`, generated `manifest.md`, tunable `stages.json`). `deploy` GETs fresh, refuses on drift (`nodes/.snapshot.json`), composes edits onto the fresh live, validates JS+structure, backs up, PUTs with a `settings` allowlist that strips `timeSavedMode` (the live GET returns it but the PUT 400s on it), reactivates, re-syncs. Stages are inferred by canvas **x-position** (the layout is already left→right) — nodes are **not** renamed because 450 `$('Name')` references + 25 patch scripts depend on current names. Expression-built prompts (`=…`: Router Agent, Bia 1/2 ESTOQUE/SEM ESTOQUE) stay in `workflow.json`; only **static** agent prompts (Memory 1/2) become `.md` files. Pure-logic modules live in [scripts/n8n/tool/](scripts/n8n/tool/) with tests (`npm run test:n8n-tool`): `tool.test.mjs` (extract/compose/snapshot/deploy_body) + `prompt-invariants.test.mjs` (locks the router/Bia/Memory output contract before any voice rewrite) + `parsers.test.mjs` (characterization + byte-fidelity + duplication-consistency net for the agent-output parsers' pure logic — color-guard, json-repair, trade-in decision; canonical blocks in [scripts/n8n/tool/parsers/blocks/](scripts/n8n/tool/parsers/blocks/), re-attach contract + duplication map in [n8n/ia-repasse-pro-v2/README.md](n8n/ia-repasse-pro-v2/README.md)). Recipe of origin: [docs/n8n-maintainability-recipe.md](docs/n8n-maintainability-recipe.md). This **coexists** with the guard + surgical patches; run the guard first either way.
- **Simulation chain (estoque + trade-in + entrada):** the sim branch is `Switch3 → CRM Leads GET Before Switch2 → Code Refresh Lead State Before Switch2 → Switch2 → Montar Body do Simulador → Simulador (crm-simulator-quote) → Parse Simulator`; the re-sim path is `Bia 2 ESTOQUE → Code Parse Re-simulacao Bia 2 ESTOQUE → Montar Body`. `lead_state` is **columnar** (one column per field, not a single jsonb), persisted via the `upsert_lead_state(text, jsonb)` RPC; text fields use `coalesce(excluded.X, prior)` (null preserves), latches use `prior OR excluded`, numerics carry-forward. Four chain bugs fixed (2026-06-15, all via surgical patches — `patch-montar-body-resolve-stockid.mjs`, `patch-montar-body-tradein.mjs`, `patch-node13-commerce-backfill.mjs`, `patch-resim-reattach-tradein.mjs`): (1) Montar Body resolves/validates `stock_item_id` against `inventory.available_items` (LLM hallucinated slugs → score by model+color+capacity, else `simulation_skipped_reason:"unresolved_stock_item"`); (2) Montar Body reads trade-in from memory/inputData/lead_state with `tradein_disqualified !== true`; (3) `Node13-Code Filtrar Resultados Estoque` back-fills `memory.desired_*` from the `Code Refresh Lead State Before Switch2` ctx root (desired_* lives at root, not under `memory`); (4) `Code Parse Re-simulacao` re-attaches trade-in/entry/card/desired fields (agent nodes drop upstream context). **Cash entry (entrada antes de simular, 2026-06-15):** before simulating, the AI asks if the client wants a down payment (Pix/dinheiro) and finances the rest on card. Three `lead_state` columns ([20260615120000_lead_state_cash_entry.sql](supabase/migrations/20260615120000_lead_state_cash_entry.sql)): `cash_entry_asked` (sticky bool), `cash_entry_intent` (bool), `cash_entry_amount` (numeric); on-conflict is coalesce-preserve so a buffer-race stale-null upsert can't wipe it. Wired across 6 layers: the RPC/columns, the routing gate in [repasse-code-routing-flags.js](scripts/n8n/repasse-code-routing-flags.js) (`needsCashEntryQuestion` / route `ask_cash_entry_before_sim`, and `cashEntryResolved` now gates `shouldSimulateNow`), the `Memory 2 - Reconciler` prompt, the `Bia 2 SEM ESTOQUE ` prompt (REGRA DE ENTRADA), the POST Lead_State builder (`Code in JavaScript`), and `Edit Fields5` (now 90 explicit assignments — it has no `includeOtherFields`, so any new persisted field must be added here too). **Buffer-race caveat:** the live smoke harness fires multiple inconsistent parallel executions per turn (a twin reprocesses an earlier message with stale state); verify behavior via the `Simulador`/`Montar Body` runData of the correct execution, not the posted reply, and rely on the RPC's coalesce-preserve for persistence safety.
- **Memory sub-pipeline (`lead_state`):** the per-turn state machine inside the workflow is `Memory 1 - Extractor → Code Parse Memory 1 → Memory 2 - Reconciler → Code Parse Memory 2 → Code in JavaScript2`; `Code in JavaScript2`'s output is persisted as `lead_state` (via `Edit Fields5 → Code in JavaScript → CRM Leads POST Lead_State`), so any corruption sticks across turns and returns as `prev`. **Ownership (changed 2026-06-14, [patch-memory2-owns-leadstate.mjs](scripts/n8n/patch-memory2-owns-leadstate.mjs)):** `Memory 2 - Reconciler` now owns the `lead_state` update — its JSON output **is** the full reconciled state (copy the prior `LEAD_STATE ATUAL`, overlay only what changed, never omit a field). `Code Parse Memory 2` is now **extraction-only** (raw code in [repasse-code-parse-memory-2.js](scripts/n8n/repasse-code-parse-memory-2.js)): robust JSON parse (markdown strip + unescaped-quote repair), no reconciliation, and **graceful** on failure (`memory.parse_error` instead of a hard throw) — it still passes the prior `lead_state` (as `prev`) and `last_message_content` through. **`Parse Memory` was removed (manual edit, 2026-06-14)** — the deterministic `preserve()` + guardrails safety net no longer exists in the live workflow; the LLM `Memory 2 - Reconciler` is now the sole authority on `lead_state`, and the trivial `Code in JavaScript2` (`item.json = $input.first().json.memory`) just flattens the reconciled `memory` object onto the root before `Edit Fields5`. Consequence: the old deterministic readiness/trade-in-consent guardrails (`REPASSE V2 MULTI QUOTE READINESS`, `REPASSE DETERMINISTIC CORE`, `can_simulate_tradein`, `tradeinOk`) are gone — any field the LLM drops is no longer back-filled from `prev`. **Gotcha:** the `@n8n/n8n-nodes-langchain.agent` nodes emit only `{ output }` and drop all upstream context, so the `Code Parse Memory *` nodes must re-attach whatever downstream nodes read off `inputData`. `Edit Fields5` rebuilds the rest of the context from `$('Edit Fields')` / `$('Edit Fields4')` / `$('Code Parse Memory 1')` (e.g. `intent_confidence` now reads `$('Code Parse Memory 1').item.json.memory_extraction.confidence`). The trade-in-vs-desired disambiguation now lives entirely in the `Memory 2 - Reconciler` prompt (it reads the last bot message); the historical guardrail patch [patch-parse-memory-tradein-state.mjs](scripts/n8n/patch-parse-memory-tradein-state.mjs) targeted the now-deleted `Parse Memory` node and is retained only for reference.

### CRM Plus (pages, access, handoff)
The CRM lives in [pages/crm/](pages/crm/) (one page per feature, lazy-loaded) plus shared UI in [components/crm/](components/crm/) (conversation list/messages panels, message bubbles, audio record/playback, AI commerce-state panel).

- **Page inventory & access** — the catalog of CRM pages, their nav section, and which `AppRole`s see them is **data-driven** from [components/crm/pageAccess.ts](components/crm/pageAccess.ts) (`CRM_PAGE_ACCESS`). Two sections:
  - **service** (admin/manager/seller): `conversations`, `comments`, `leads`, `funnels`, `statistics`, `ads`, `forms` (+ `simulator`, admin/seller only).
  - **admin** (admin only): `ai-settings` (Agentes IA), `automations`, `broadcasts`, `templates`, `custom-fields`, `attendance-scripts`, `integrations`, `cashback`, `settings`.
  Adding a CRM page means extending the `CRMPage` union + `CRM_PAGE_ACCESS` here, plus icon/title in [components/crm/crmPageMeta.ts](components/crm/crmPageMeta.ts), the route in [CRMStandaloneApp](components/crm/CRMStandaloneApp.tsx), and the path map in [lib/crmRouting.ts](lib/crmRouting.ts). Default landing page is `conversations`.
- **Cross-host SSO** — to open the standalone CRM already authenticated, the app mints a one-time code and the standalone app consumes it via the `crm-auth-handoff` edge function (`?handoff=<code>` → exchanges for a Supabase session). See `CRMHandoffBootstrap` in `CRMStandaloneApp.tsx`.
- **Realtime** — `ConversationsPage` subscribes to `postgres_changes` on `crm_conversations` to keep the conversation list live; sends go through `crm-send-message` and message actions through `crm-uaz-message-action`.
- **AI handoff UI** — `transferencia_pendente` vs `em_atendimento_humano` (see AI pipeline above) are surfaced in `ConversationsPage`; the AI commerce/trade-in snapshot (missing fields, pending actions) renders via [AICommerceStatePanel](components/crm/AICommerceStatePanel.tsx) from [lib/crm/aiCommerceSnapshot.ts](lib/crm/aiCommerceSnapshot.ts) + [lib/crm/commerceState.ts](lib/crm/commerceState.ts).
- **CRM edge functions** — `crm-send-message`, `crm-ai-inbound`, `crm-uaz-webhook-receiver`, `crm-instagram-webhook-receiver`, `crm-uaz-message-action`, `crm-uaz-media-download`, `crm-uaz-instance-admin`, `crm-audio-transcribe`, `crm-auth-handoff`, `crm-conversation-handoff`, `crm-delete-conversation`, `crm-leads-api`, `crm-lead-profile`, `crm-simulator-quote`, `crm-broadcast-worker`, `crm-scheduled-messages-worker`, `crm-event-publisher`, `crm-n8n-api`, `crm-ai-agent-test-endpoint`.

### Database
88+ ordered SQL files in [supabase/migrations/](supabase/migrations/) (timestamped `YYYYMMDDHHMMSS_*.sql`). This is the source of truth for the schema; add a new migration rather than editing existing ones.

## Conventions

- Path alias `@/` → repo root (configured in `tsconfig.json`, `vite.config.ts`, and both Vitest configs).
- Tests are colocated (`Foo.test.tsx` next to `Foo.tsx`), Testing Library + jsdom; global setup in [tests/setup.ts](tests/setup.ts). Some tests use a `.red.test.tsx` suffix for TDD red-phase specs.
- `noUnusedLocals`/`noUnusedParameters` are on — remove dead bindings. ESLint additionally enforces dead/unreachable-code guards (`no-unreachable`, `no-fallthrough`, etc.); see [eslint.config.js](eslint.config.js) for why.
- Build is manually chunked (`vendor-*` groups) with a 500 kB chunk warning limit — keep an eye on bundle size when adding heavy deps.
- PWA service worker is `public/sw.js` via `vite-plugin-pwa` `injectManifest`; the CRM and main app have separate web manifests.
- Mobile UI follows Apple HIG conventions documented in [AgenteHIG.md](AgenteHIG.md). The shared visual layer (design tokens, motion, theming) lives in [design-system/seroclub-iphonerepasse/](design-system/seroclub-iphonerepasse/).
- Implementation plans/specs live in three places: hand-written PRDs in [tasks/](tasks/) (`prd-*.md`, Portuguese), skill-generated plans/specs in [docs/superpowers/plans/](docs/superpowers/plans/) and `docs/superpowers/specs/`, and `prd.json` task lists in [ralph/](ralph/) for the Ralph autonomous-agent loop.
